## 1. Complete Journey (happy path)

```
Sign in
   │
   ▼
Dashboard ──────────────► New Project
                              │
                              ▼
                    Step 1 · INPUT
          (Paste JSON | Schema Builder | Swagger)
                              │
                              ▼
                    Step 2 · CONFIGURE
        Validation? ☑ Zod ☑ Yup  (Advanced: ☐ JSON Schema)
        Types?      ☑ TypeScript
        Methods?    ☑ GET ☑ POST ☐ PUT ☐ PATCH ☐ DELETE
        Mock records per entity: [25]
                              │
                              ▼
                    Step 3 · REVIEW
        Parsed Internal Project Schema (IPS) tree
        Inline edits · validation rules visible
                              │
                        [ Generate ]
                              │
                              ▼
                    Step 4 · PROGRESS
        Live worker board (only selected workers)
                              │
                              ▼
                    Step 5 · SUCCESS / PROJECT PAGE
        Hosted API URL · Artifact grid · Download ZIP
```

Target: the whole path completes in **under 3 minutes**.

## 2. Step-by-Step

### Step 0 · Dashboard
- Project cards: name, status (`draft · generating · active · expired`), expiration countdown, plan badge.
- Primary CTA: **New Project**. Expired cards show **Generate Again**.

### Step 1 · Input
The user picks one input source:

| Tab | Behavior |
|---|---|
| **Paste JSON** | Paste a sample payload → parser infers entities, field types, and likely formats (email/phone/url by key name) |
| **Manual Schema Builder** | Add entities and fields by hand; unlimited nesting via object/array field types; dynamic arrays with "Add Another"; per-field validation popover |
| **Swagger / OpenAPI** | Upload spec file → parsed into the IPS |

All three converge to the same place: an **IPS draft**.

### Step 2 · Configuration questions
The system asks, in order:

1. **Validation needed?** — checkboxes ☑ Zod ☑ Yup; "Advanced" expander reveals ☐ JSON Schema.
2. **Schema types?** — ☑ TypeScript.
3. **Which API methods?** — checkboxes GET / POST / PUT / PATCH / DELETE.
   - Helper note: *"Seed data is always generated for readable endpoints. A GET-only API is read-only."*
4. **Mock records per entity** — default 25.

These answers form the **Generation Config** stored on the job; only matching workers will run.

### Step 3 · Review
- Left: collapsible IPS tree — entities → fields → nested structures → validation rules (Layer 1 auto-detected rules shown as chips, Layer 2 custom rules editable).
- Right: generation summary — "Will generate: Zod, Yup, TypeScript, 25 × 2 entities mock records, OpenAPI, Postman, Hosted API (GET, POST), Export ZIP."
- Inline edits allowed. **Generate** is the only way forward — review is mandatory.

### Step 4 · Progress
- Vertical worker board; each row: `pending → generating (pulse) → ✓ completed` or `❌ failed + [Retry]`.
- Only workers implied by the Generation Config appear.
- Overall progress bar; page is safe to leave (generation continues in background; user is notified on completion).

### Step 5 · Success / Project Page
- **Hosted API card:** URL (e.g., `https://api.InstantMockAPI.dev/p/{projectId}/customers`), copy button, method chips, mini request playground.
- **Artifact grid:** one card per artifact with status, version, timestamp, and View / Download / Regenerate actions.
- **Download ZIP** and **Regenerate** in the header; version history panel with Restore.

## 3. Decision Tree

```
Does the user have a sample JSON payload?
├── Yes → Paste JSON → parser
└── No
    ├── Has a Swagger/OpenAPI file? → Upload → parser
    └── Neither → Manual Schema Builder

Which validators?            → run Worker B with selected targets only
TypeScript selected?         → run Worker C, else skip
Any HTTP method selected?    → run Worker F (hosted API) + Worker D (seed data)
No methods selected?         → skip Worker F; Worker D still runs (mock data is a core downloadable artifact)
Export always?               → Worker G runs when any downloadable artifact was produced
```

## 4. Failure & Retry Flow (per-worker)

```
Progress board:
  Schema      ✓
  Validation  ✓
  Types       ✓
  Mock Data   ❌  "Faker seed error…"   [Retry]
  Hosted API  ⏳ waiting on Mock Data
  Export      ⏳

User clicks Retry → ONLY Worker D re-runs → downstream workers resume.
Completed artifacts are never re-generated by a retry.
The project as a whole never enters a failed state.
```

## 5. Expiry & Regeneration Flow

```
Active project
   │  (plan lifetime elapses: Free 2d · Pro 7d · Enterprise 30d)
   ▼
Pre-expiry reminder (email + dashboard) → "Download ZIP" nudge
   │
   ▼
EXPIRED
   • Hard-deleted: hosted API + URL, generated files, mock data, caches
   • Kept: project name, input, IPS, generation config, version history
   │
   ▼
Project page shows: grayed hosted-API card, "Hosted assets deleted",
                    single CTA: [ Generate Again ]
   │
   ▼
Generate Again → new Generation Job (same config) → Progress → new hosted URL
```

One click. Nothing re-entered.

## 6. Per-Asset Regeneration Flow

```
Project page → Regenerate → modal:
   ☑ Zod   ☐ Types   ☐ Mock Data   ☐ OpenAPI   ☐ Hosted API …
→ enqueues only the selected workers
→ each regenerated artifact gets a new version entry
```

## 7. Versioning Flow

```
Generation #1  → all artifacts stamped v1
User edits schema in Review → Generate → v2
Version history panel → [Restore v1] → IPS + config revert; next generation stamps v3 from v1's model
```

## 8. Concurrency Guard

```
User clicks Generate while a job is active:
  Free (limit 1)      → new request is queued behind the running job, with a visible "queued" state
  Pro (limit 3)       → runs in parallel up to 3; 4th queues
  Enterprise          → runs immediately
Duplicate rapid clicks on the same project → idempotent: one job, never N.
```
